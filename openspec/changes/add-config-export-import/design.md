## Context

The webapp already holds a device's full configuration in memory the moment
you're connected: the layer/button JSON (`config`) and every icon bitmap
it's downloaded or generated (`Editor.icons`, a `Map<iconId, {bytes, previewCanvas}>`
- see `add-image-buttons`). Pushing that state to a device already goes through
`webapp-ble-sync`'s upload-then-push flow, which uploads any icon the *connected*
device doesn't yet have (tracked via a per-session `deviceIconIds` set in
`webapp/js/app.js`) before sending the config itself. Restoring a config onto a
brand-new device is really just "make that same in-memory state available again,
on a different device, in a different browser session" - the missing piece is
purely getting it in and out of the browser as a file, since nothing today
persists it anywhere outside of one page session and one physical device's flash.

## Goals / Non-Goals

**Goals:**
- Export the currently-loaded configuration and every icon bitmap the editor
  currently has bytes for into one downloadable JSON file.
- Import such a file back into the editor's in-memory state - config and icon
  previews - without requiring any device to be connected.
- Restoring a backup and then pushing it to a device (new or old) works through
  the exact push flow that already exists, unmodified.

**Non-Goals:**
- Automatic/scheduled backups, cloud storage, or any server-side component - this
  stays a static webapp with local file download/upload only, consistent with the
  project's no-backend architecture.
- Backup file encryption, signing, or access control - same local-trust model as
  everything else here (Web Bluetooth pairing, LittleFS, the SD mirror).
- Multiple named backup profiles, a backup history/version list, or diffing
  between backups - one export action produces one file; the user manages files
  themselves on their own filesystem.
- Migrating a backup written under one config schema version to a newer one -
  mismatches are surfaced to the user the same way a live device's schema
  mismatch already is (see `webapp-ble-sync`'s existing confirm-to-proceed
  behavior), not silently transformed.

## Decisions

### Backup file is a single JSON document: config + a map of icon id -> base64 bytes
```json
{
  "backupFormatVersion": 1,
  "config": { "schemaVersion": 1, "rootLayer": "root", "layers": { ... } },
  "icons": { "a3f9c21b": "<base64 of 8192 RGB565 bytes>", ... }
}
```
- `backupFormatVersion` is a new, independent version number for this file's own
  shape (separate from `config.schemaVersion`, which already versions the config
  JSON itself) - so the backup *container* format can evolve later without
  conflating it with config schema changes.
- Icons are embedded as base64 text rather than a separate archive/zip format -
  keeps this a single plain-text file (easy to inspect, diff, or hand-edit) and
  needs no new dependency for either encoding or decoding; base64 overhead
  (~33%) on an 8192-byte icon is a non-issue at this scale.
- Only icons actually *referenced by the current configuration* are included -
  export filters `Editor.icons` down to `collectIconIds(config)` (the same helper
  `webapp-ble-sync` already uses to decide what to pull/push), not the raw
  `Editor.icons` Map as-is. `Editor.icons` accumulates entries across the whole
  page session - every device connected to, every template applied, every icon
  ever uploaded or cleared - and nothing ever prunes it, so serializing it
  unfiltered would silently bundle stale bytes with nothing to do with the
  config being exported. If a config references an icon id the editor never
  downloaded or generated bytes for, it's simply absent from the backup's
  `icons` map, exactly as if it were a never-uploaded icon on restore: falls
  back to the slot's color, per `macropad-display-ui`'s existing missing-icon
  fallback. No new failure mode, no orphan-tracking added.
- Alternative considered: a zip containing the config JSON plus one `.bin` file
  per icon. Rejected - needs a zip library (new dependency) for marginal benefit
  over base64-in-JSON at these icon counts/sizes, and a flat JSON file is easier
  for a user to eyeball or hand-edit if something goes wrong.

### Export/import work entirely on in-memory editor state, no device required
`Export` serializes whatever `config` and `Editor.icons` currently hold -
regardless of whether that state came from a live device pull, mid-edit, or a
previous restore. `Import` deserializes into those same shapes and calls
`Editor.setConfig()`, exactly mirroring what happens after a BLE pull. This means
importing a backup with no device connected still lets you review/edit the
layout and see icon previews, and pushing it later goes through the *existing*
`webapp-ble-sync` upload-before-push logic completely unchanged: a freshly
connected device's `deviceIconIds` is empty, so every icon in the restored config
gets uploaded automatically, same as any first-time config push.

**Important implementation detail, not just an `Editor` concern**: `app.js` keeps
its own module-level `config` binding, separate from whatever `Editor` does
internally - the existing connect handler already has to do two things on a
pull (`config = await connection.pullConfig(); editor.setConfig(config);`), and
`Editor.setConfig()` does NOT call back into `onChange`/update that outer
binding. The restore handler must follow that exact same two-step pattern
(reassign `app.js`'s own `config`, then call `editor.setConfig(config)`) -
calling only `Editor.setConfig()` would update what's visually shown but leave
`pushBtn`'s handler still reading the stale previous `config`, silently pushing
the wrong thing. This is the one place restore ISN'T a simple drop-in call to
an `Editor` method; task 3.2 spells this out explicitly for exactly that reason.

Restoring also leaves the "Connected and up to date with the device" status
line (if currently shown) inaccurate - the editor state no longer reflects what
the connected device actually has until the next push. The restore handler
should update the status text to reflect that a restore just happened and
hasn't been pushed yet.
- Alternative considered: tie "restore" directly to "push to this connected
  device" as one combined action. Rejected - forces a device to be present just
  to *look at* a backup, and duplicates upload/push logic that already exists
  and is already tested; decoupling restore-into-editor from push-to-device
  keeps each concern in exactly one place.

### Per-icon size validation on import, mirroring the firmware's own validation
Before accepting an icon's decoded bytes from a backup file into `Editor.icons`,
the webapp checks it's exactly `ICON_BYTES` (8192) long - the same check the
firmware itself applies before saving an uploaded icon (`macropad-icon-assets`).
An icon entry that fails this check is dropped (with the user told which icon id
was skipped) rather than accepted and later silently rejected by the device mid
-push. This keeps validation close to where bad data would otherwise surface,
instead of only discovering it during a live upload.

## Risks / Trade-offs

- **Icon-less recovery for an icon id missing from the backup** (see Decision 1)
  → mitigated by the same graceful color-fallback that already exists for a
  missing/invalid icon on-device; not a new failure mode.
- **Hand-edited or corrupted backup files** (malformed JSON, wrong-length icon
  bytes, unknown `backupFormatVersion`) → mitigated by validating the config
  shape the same way a BLE pull's JSON is already handled, and by the per-icon
  size check above; a bad file produces a clear error instead of a broken editor
  state.
- **No versioned migration path if `backupFormatVersion` ever needs to change
  shape** → acceptable at `backupFormatVersion: 1`; revisit if/when a version 2
  actually becomes necessary, consistent with this project's general "don't
  build for hypothetical future requirements" approach.
- **A completely unparseable/malformed backup file** (not just a wrong-length
  icon or a schema mismatch, but invalid JSON or missing required fields
  entirely) → `deserializeBackup` throws rather than returning a best-effort
  partial result; the restore-file-input handler catches this and shows the
  user a clear error, leaving whatever was already loaded in the editor
  untouched (see the new spec requirement for this).
- **Object URL lifecycle on export** → the download flow (Blob + object URL +
  synthetic `<a download>` click) must call `URL.revokeObjectURL()` once the
  download has been triggered, or each export leaks the blob for the rest of
  the page's lifetime.
